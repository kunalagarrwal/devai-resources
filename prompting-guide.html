Overview

The Prompt Engineering Guide is a free, community-maintained reference covering techniques for writing prompts that produce reliable, well-structured outputs from LLMs. Unlike the API docs (which explain how to call a model), this guide explains how to communicate with it effectively — start here before you touch any API.

Every prompt has four components — missing any one causes failures:

📌

Instruction

What you want the model to do. Be specific — vague instructions produce vague outputs.

🗂️

Context

Background the model needs to answer correctly — user role, app purpose, constraints.

📄

Input Data

The actual content to process — the user's message, the document, the review text.

📐

Output Format

How you want the response structured — JSON, markdown, a numbered list, a single word.

What's in it?

The guide is a progressive curriculum from fundamentals to advanced techniques. Here's what you'll actually use when building an app:

TechniqueHow it worksBest for2026 note
Zero-shot Give the task with no examples — rely on the model's training. General instructions, simple classifications Works well on capable models (GPT-5.4, Sonnet 4.6)
Few-shot highest impact Provide 2–5 input → output examples before the actual task. Structured outputs (JSON, tables), consistent formatting Still the single most effective technique for format control
Chain-of-thought (CoT) Add "Think step by step" — model reasons out loud before answering. Math, logic, multi-step analysis Skip on reasoning models (o3, Claude Extended Thinking) — they already do this internally
Self-consistency Generate multiple answers, take the majority vote. Hard reasoning where one pass isn't reliable Effective on non-reasoning models; less needed on o3/o4-mini
ReAct Interleave reasoning steps with tool-use actions. Agentic workflows (search, code, APIs) Core pattern behind LangGraph and OpenAI Responses API agents

Relevance to Your Audience

Getting the API to work is only half the challenge. Getting it to produce reliable, correctly formatted output is the other half — and that's entirely a prompting problem.

  • 🐛
    Prompt failures look like bugs If your JSON extractor sometimes returns plain text, or your classifier sometimes adds unprompted explanation — that's a prompt structure issue, not a model capability issue. This guide gives you the vocabulary to diagnose it.
  • ✨
    Few-shot examples fix most inconsistency Adding 2–3 input/output examples to your system prompt is often the difference between a flaky feature and a reliable one. Try it before anything else when your output isn't consistent.
  • 🧮
    Know which model type you're prompting Standard models (GPT-5.4, Claude Sonnet) benefit from CoT. Reasoning models (o3, o4-mini, Claude with Extended Thinking) already think internally — adding CoT hurts more than it helps. Specify the output format instead.
  • 🔄
    Return to this guide when a feature breaks Whenever something doesn't work right, the root cause is almost always the prompt. This guide helps you identify whether you're missing context, need examples, or need a tighter output specification.

2025: Know which model type you're prompting

The single most important shift in prompt engineering in 2025 is understanding the difference between standard models and reasoning models. For standard models (GPT-5.4, Claude Sonnet 4.6, Gemini Flash), chain-of-thought prompting ("think step by step") dramatically improves results on complex tasks. For reasoning models (o3, o4-mini, Claude with Extended Thinking), the model already does extensive internal reasoning — adding CoT instructions is unnecessary and can hurt performance by interrupting the model's natural thinking process. Instead, give reasoning models clear problem descriptions and precise output requirements, and let them figure out the reasoning path themselves.

Tips for using this resource effectively

  • Read the Introduction and Basics sections even if you've used LLMs before — the guide formalizes concepts you may have discovered informally and introduces precise terminology (zero-shot, few-shot, CoT) that's useful both for reasoning about your work and for communicating with colleagues about it.
  • Few-shot prompting is the most immediately applicable technique. If you need the model to return data in a specific format — JSON, markdown tables, numbered lists, anything structured — include 2–3 complete input/output examples in your system prompt. The improvement in consistency is almost always dramatic.
  • Use chain-of-thought prompting for standard models (GPT-5.4, Claude Sonnet 4.6) when you need reasoning, comparison, or multi-step analysis. However, skip CoT for reasoning models (o3, o4-mini, Claude with Extended Thinking) — they already reason internally and your instructions are better spent specifying the output clearly.
  • The guide includes runnable prompt examples you can test in the OpenAI Playground or any model interface. Don't just read them — copy them, run them, and modify them. Hands-on experimentation with the examples is far more instructive than reading alone.
  • Return to this guide whenever a feature in your application isn't behaving as expected. The underlying issue is almost always the prompt structure, not the model's capability. This resource will help you diagnose whether you're missing context, need examples, or need a different output format specification.

Few-Shot Prompting: Before & After

Few-shot prompting is the most impactful technique for structured outputs. This Python example shows how adding just two examples to a system prompt transforms a flaky response into a reliable, parseable one.

from openai import OpenAI client = OpenAI() # ❌ BEFORE: vague instruction produces inconsistent output bad_system = "Extract the sentiment from product reviews." # AFTER: few-shot examples lock in the exact output format good_system = """Extract the sentiment from product reviews. Return ONLY a JSON object with this exact structure. Example 1: Review: "Works great, exactly as described!" Output: {"sentiment": "positive", "confidence": "high"} Example 2: Review: "Stopped working after two days, very disappointed." Output: {"sentiment": "negative", "confidence": "high"} Example 3: Review: "It's okay I guess, nothing special." Output: {"sentiment": "neutral", "confidence": "medium"} Now classify the review the user sends.""" def classify_review(review_text): response = client.chat.completions.create( model="gpt-5.4-mini", messages=[ {"role": "system", "content": good_system}, {"role": "user", "content": review_text} ], temperature=0 # temperature=0 maximizes consistency for structured tasks ) return response.choices[0].message.content print(classify_review("Battery lasts forever, super happy with this purchase!")) # Output: {"sentiment": "positive", "confidence": "high"}
